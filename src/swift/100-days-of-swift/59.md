### Key points

::: details Key points

[üìélink](https://www.hackingwithswift.com/guide/6/2/key-points)

here are three pieces of code I‚Äôd like to revisit briefly, just to make sure you understand them fully.

First, I want to look more closely at how closure capturing works with `asyncAfter()`. Here‚Äôs some code as an example:

```swift
DispatchQueue.main.asyncAfter(deadline: .now() + 1) { [unowned self] in
    self.doStuff()
}
```

The call to `asyncAfter()` needs two parameters, but we pass the second one in using trailing closure syntax because it‚Äôs clearer. The first parameter is specified as a `DispatchTime` value, which is the exact time to execute the code. When we specify `.now()`, Swift is smart enough to realize that means `DispatchTime.now()`, which is the current time. It then lets us add 1 second to it, so that the finished deadline ends up being one second away from now.

Then there‚Äôs the `[unowned self]`. This is called a capture list, and it gives Swift instructions how to handle values that get used inside the closure. In this case, `self.doStuff()` references `self`, so the capture list `[unowned self]` means ‚ÄúI know you want to capture `self` strongly so that it can be used later, but I want you not to have any ownership at all.‚Äù

If the closure runs and `self` has become nil for some reason, the call to `self.doStuff()` will crash: we told Swift not to worry about ownership, then accidentally let `self` get destroyed, so the crash is our own fault. As an alternative, we could have written `[weak self]`, which would capture `self` in the closure as an optional. With that change, you‚Äôd need to run this code instead:

```swift
self?.doStuff()
```

Closure capturing can be a complicated topic. In this case, `[unowned self]` isn‚Äôt really even needed because there‚Äôs no chance of a reference cycle: `self` doesn‚Äôt own `DispatchQueue.main`, so the reference will be destroyed once the closure finishes. However, there‚Äôs no harm adding `[weak self]`, which is why I often include it.

The second piece of code I‚Äôd like to review is this, taken from project 15:

```swift
UIView.animate(withDuration: 1, delay: 0, options: [], animations: {
    switch self.currentAnimation {
    case 0:
        break

    default:
        break
    }
}) { finished in
    sender.isHidden = false
}
```

At this point in your Swift career you‚Äôve seen several functions that accept closures, but this one takes two: one is a set of animations to perform, and one is a set of actions to run when the animations have completed. I didn‚Äôt include closure capturing here because it isn‚Äôt needed ‚Äì the animation closure will be used once then destroyed.

I don‚Äôt want to sound like a broken record, but: if you use `[weak self]` when it isn‚Äôt needed, nothing happens. But if you don‚Äôt use it and it was needed, bad things will happen - at the very least you‚Äôll leak memory.

The final piece of code to review is this, also taken from project 15:

```swift
self.imageView.transform = CGAffineTransform(rotationAngle: CGFloat.pi)
self.imageView.transform = CGAffineTransform.identity
```

There are a few things in there that I‚Äôd like to recap just briefly:

1. The `self` is required when accessing `imageView`, because we‚Äôre inside a closure and Swift wants us to explicitly acknowledge we recognize `self` will be captured.
2. The `CGFloat.pi` constant is equal to 180 degrees, and it also has equivalents in `Float.pi` and `Double.pi` ‚Äì this is just to save you having to typecast values.
3. The identity matrix, `CGAffineTransform.identity`, has no transformations: it‚Äôs not scaled, moved, or rotated, and it‚Äôs useful for resetting a transform.

### Challenge

::: details Challenge

[üìélink](https://www.hackingwithswift.com/guide/6/3/challenge)

Your challenge is to make an app that contains facts about countries: show a list of country names in a table view, then when one is tapped bring in a new screen that contains its capital city, size, population, currency, and any other facts that interest you. The type of facts you include is down to you ‚Äì Wikipedia has a huge selection to choose from.

To make this app, I would recommend you blend parts of project 1 project 7. That means showing the country names in a table view, then showing the detailed information in a second table view.

How you load data into the app is going to be an interesting problem for you to solve. I suggested project 7 above because a sensible approach would be to create a JSON file with your facts in, then load that in using `contentsOf` and parse it using `Codable`. Regardless of how you end up solving this, I suggest you _don‚Äôt_ just hard-code it into the app ‚Äì i.e., typing all the facts manually into your Swift code. You‚Äôre better than that!

Go ahead and try coding it now. If you hit problems, here are some hints:

- You should create a custom `Country` struct that has properties for each of the facts you have in your JSON file. You can then have a `[Country]` array in your view controller.
- When using a table view in your detail view controller, try setting the `numberOfLines` property of the cell‚Äôs text label to be 0. That ought to allow the cell to fill up to two lines of text by default.
- Don‚Äôt forget all the little UI touches: adding a disclosure indicator to the countries table, adding titles to the navigation controller, and so on. You could even add an action button to the detail view that shares a fact about the selected country.
