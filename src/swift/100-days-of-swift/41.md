There are three Swift features that are so important ‚Äì and so commonly used ‚Äì that they are worth revising to make sure you‚Äôre comfortable with them.

The first piece of code I‚Äôd like to look at is this one:

```swift
for (index, line) in lines.enumerated() {
    let parts = line.components(separatedBy: ": ")
```

This might seem like a new way of writing a loop, but really it‚Äôs just a variation of the basic `for` loop. A regular `for` loop returns one value at a time from an array for you to work with, but this time we‚Äôre calling the `enumerated()` method on the array, which causes it to return _two_ things for each item in the array: the item‚Äôs position in the array, as well as the item itself.

It‚Äôs common to see `enumerated()` in a loop, because its behavior of delivering both the item and its position is so useful. For example, we could use it to print out the results of a race like this:

```swift
let results = ["Paul", "Sophie", "Lottie", "Andrew", "John"]

for (place, result) in results.enumeradidSet. This meant that whenever the scorted() {
    print("\(place + 1). \(result)")
}
```

Note that I used `\(place + 1)` to print out each person‚Äôs place in the results, because array positions all count from 0.

The second piece of code we‚Äôre going to review is this:

```swift
var score: Int = 0 {
    didSet {
        scoreLabel.text = "Score: \(score)"
    }
}
```

This is a property observer, and it means that whenever the `score` integer changes the label will be updated to match. There are other ways you could ensure the two remain in sync: we could have written a `setScore()` method, for example, or we could just have updated the `scoreLabel` text by hand whenever the `score` property changed.

The former isn‚Äôt a bad idea, but you do need to police yourself to ensure you never set `score` directly - and that‚Äôs harder than you think! The second _is_ a bad idea, however: duplicating code can be problematic, because if you need to change something later you need to remember to update it everywhere it‚Äôs been duplicated.

The final piece of code I‚Äôd like to look at again is this:

```swift
DispatchQueue.global().async { [weak self] in
    // do background work

    DispatchQueue.main.async {
        // do main thread work
    }
}
```

That code uses Grand Central Dispatch to perform some work in the background, then perform some more work on the main thread. This is _extremely_ common, and you‚Äôll see this same code appear in many projects as your skills advance.

The first part of the code tells GCD to do the following work on a background thread. This is useful for any work that will take more than a few milliseconds to execute, so that‚Äôs anything to do with the internet, for example, but also any time you want to do complex operations such as querying a database or loading files.

The second part of the code runs after your background work has completed, and pushes the remaining work back to the main thread. This is where you present the user with the results of your work: the database results that matched their search, the remote file you fetched, and so on.

It is _extremely_ important that you only ever update your user interface from the main thread ‚Äì trying to do it from a background thread will cause your app to crash in the best case, or ‚Äì much worse ‚Äì cause weird inconsistencies in your app.

This is the first challenge that involves you creating a game. You‚Äôll still be using UIKit, though, so it‚Äôs a good chance to practice your app skills too.

The challenge is this: make a hangman game using UIKit. As a reminder, this means choosing a random word from a list of possibilities, but presenting it to the user as a series of underscores. So, if your word was ‚ÄúRHYTHM‚Äù the user would see ‚Äú??????‚Äù.

The user can then guess letters one at a time: if they guess a letter that it‚Äôs in the word, e.g. H, it gets revealed to make ‚Äú?H??H?‚Äù; if they guess an incorrect letter, they inch closer to death. If they seven incorrect answers they lose, but if they manage to spell the full word before that they win.

That‚Äôs the game: can you make it? Don‚Äôt underestimate this one: it‚Äôs called a challenge for a reason ‚Äì it‚Äôs supposed to stretch you!

The main complexity you‚Äôll come across is that Swift has a special data type for individual letters, called `Character`. It‚Äôs easy to create strings from characters and vice versa, but you do need to know how it‚Äôs done.

First, the individual letters of a string are accessible simply by treating the string like an array ‚Äì it‚Äôs a bit like an array of `Character` objects that you can loop over, or read its `count` property, just like regular arrays.

When you write `for letter in word`, the `letter` constant will be of type `Character`, so if your `usedLetters` array contains strings you will need to convert that letter into a string, like this:

```swift
let strLetter = String(letter)
```

Note: unlike regular arrays, you can‚Äôt read letters in strings just by using their integer positions ‚Äì they store each letter in a complicated way that prohibits this behavior.

Once you have the string form of each letter, you can use `contains()` to check whether it‚Äôs inside your `usedLetters` array.

That‚Äôs enough for you to get going on this challenge by yourself. As per usual there are some hints below, but it‚Äôs always a good idea to try it yourself before reading them.

- You already know how to load a list of words from disk and choose one, because that‚Äôs exactly what we did in tutorial 5.
- You know how to prompt the user for text input, again because it was in tutorial 5. Obviously this time you should only accept single letters rather than whole words ‚Äì use `someString.count` for that.
- You can display the user‚Äôs current word and score using the `title` property of your view controller.
- You should create a `usedLetters` array as well as a `wrongAnswers` integer.
- When the player wins or loses, use `UIAlertController` to show an alert with a message.

Still stuck? Here‚Äôs some example code you might find useful:

```swift
let word = "RHYTHM"
var usedLetters = ["R", "T"]
var promptWord = ""

for letter in word.characters {
    let strLetter = String(letter)

    if usedLetters.contains(strLetter) {
        promptWord += strLetter
    } else {
        promptWord += "?"
    }
}

print(promptWord)
```
